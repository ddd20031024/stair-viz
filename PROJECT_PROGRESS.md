# 爬楼助力机器人上位机项目进度说明

## 1. 项目定位

本项目是爬楼助力机器人系统中的上位机模块。当前机器人下位机第一版已经完成，机械结构采用由三个小轮组成的三角轮机构，用于辅助机器人在楼梯场景下完成越障和爬升动作。

上位机当前主要承担开发测试、数据读取、状态显示和辅助决策任务。它通过串口与 STM32 下位机通信，读取 TOF 测距、IMU 姿态、电机反馈、三角轮状态等信息，并在 Web 页面中实时展示。后续上位机还将根据传感器数据判断当前场景，并向下位机下发模式、目标速度、三角轮目标角度等高层控制参数。

当前阶段的目标不是直接实现完整自主爬楼，而是先完成可靠通信、数据可视化、手动动作验证和安全边界确认，为后续半自动状态机和 ROS2/C++ 决策层迁移打基础。

## 2. 当前阶段

当前项目处于第二阶段：手动动作测试与上下位机联调。

这一阶段的核心目标是打通“上位机读取数据 -> 上位机判断状态 -> 上位机下发目标参数 -> 下位机安全执行”的基础链路。当前重点不是让 Python 上位机直接控制每个轮子的底层动作，而是验证上位机能否稳定获取真实机器人状态，并以安全的方式向下位机发送高层指令。

### 当前已完成或正在完成的工作

| 模块 | 当前进度 | 说明 |
| --- | --- | --- |
| 串口通信 | 已完成基础对齐 | 已对照可正常读数的 `robot_win` 程序，修正 CRC 初值、寄存器地址和读取方式 |
| 传感器读取 | 联调中 | 正在验证 TOF、IMU、电机反馈、三角轮状态等寄存器能否稳定读取 |
| Web 仪表盘 | 已具备基础功能 | 可通过浏览器实时显示连接状态、TOF、IMU、电机、三角轮和事件日志 |
| 调试接口 | 已加入 | 可通过 `/debug/state` 查看串口状态、回包数量、最近请求和超时信息 |
| 模拟模式 | 已具备 | 无真实硬件时可使用模拟数据源验证前端和数据链路 |
| 手动控制 | 下一步重点 | 准备增加或完善上位机向下位机下发模式、目标速度、三角轮目标角度等基础指令 |

### 当前重点任务

- 确认 COM 串口连接稳定，真实下位机能够持续响应上位机读请求。
- 确认逐个寄存器读取方式可靠，避免因地址跳号导致批量读取错误。
- 确认 Web 页面能稳定显示 TOF、IMU、电机反馈和三角轮状态。
- 增加手动动作测试能力，例如模式切换、低速前进、三角轮目标角度设置、停止和急停。
- 明确上位机下发的是模式和目标参数，而不是直接承担底层电机闭环。

### 当前不做的内容

- 不由 Python 上位机承担电机电流、速度或位置的底层闭环。
- 不让 Python 上位机执行毫秒级轮胎动作控制。
- 不把急停、失稳保护、指令超时保护唯一依赖放在上位机。
- 不直接进入完全自主爬楼阶段。

## 3. 系统架构分工

为了兼顾开发效率、实时性和安全性，系统采用三层分工。

| 层级 | 推荐技术 | 主要职责 |
| --- | --- | --- |
| STM32 下位机 | C/C++ | 电机实时控制、安全保护、急停、指令超时降级、三角轮执行、速度/扭矩闭环 |
| Python 上位机 | Python + FastAPI + WebSocket | 开发调试、传感器读取、Web 可视化、数据记录、手动控制、状态机原型 |
| 未来决策层 | Linux + C++/ROS2 | 稳定后的场景判断、策略决策、传感器融合、状态机部署 |

这种分工的核心原则是：上位机负责判断“当前是什么场景、下一步目标是什么”，下位机负责保证“动作如何实时、安全、稳定地执行”。

推荐的控制链路如下：

```text
传感器 / 电机 / 三角轮
        ↓
STM32 下位机实时控制层
        ↑↓ 串口或未来 CAN
Python 上位机调试与状态机原型
        ↑↓ WebSocket / 日志 / 未来 ROS2 Topic
Web 仪表盘与调试工具
```

## 4. 第二阶段任务拆分

### 4.1 通信与数据稳定性

- 对齐下位机协议，保证读请求、写请求、CRC 和寄存器地址与当前固件一致。
- 读取真实机器人数据，确认 TOF、IMU、电机反馈、三角轮反馈能够持续更新。
- 保留调试接口，持续观察 `total_responses`、`crc_errors`、`timeout_count` 等通信状态。
- 对串口异常、CRC 错误、无回包等情况进行明确提示。

### 4.2 Web 可视化与调试

- 保持现有 Web 仪表盘作为主要调试入口。
- 显示机器人连接状态、底盘模式、TOF 最小距离、IMU 姿态、电机在线状态、三角轮角度。
- 增加必要的状态提示，例如传感器失效、串口超时、下位机未连接。
- 后续增加实验数据记录入口，便于调参和复盘。

### 4.3 手动动作测试

- 在上位机加入基础命令面板，用于手动下发模式和目标参数。
- 初期命令建议包括：
  - 安全模式 / 开环模式 / 闭环模式 / 从机模式切换
  - 低速前进和停止
  - 三角轮目标角度设置
  - 三角轮占空比或角度控制模式切换
  - 急停和恢复
- 所有手动命令必须配合下位机保护逻辑验证，不能只验证上位机是否成功发送。

### 4.4 安全边界验证

- 上位机断连时，下位机应进入安全状态或保持可控降级状态。
- 指令超时时，下位机不应继续执行旧目标。
- IMU 姿态异常、TOF 数据异常、电机反馈异常时，应能触发保护或提示。
- 上位机只能作为辅助决策来源，不能作为唯一安全保护来源。

## 5. 后续计划

### 第三阶段：半自动状态机

在第二阶段手动动作验证稳定后，进入半自动状态机开发。该阶段目标是让上位机根据传感器数据判断机器人所处阶段，并给下位机下发高层目标参数。

建议状态如下：

```text
IDLE
APPROACH_STAIR
PREPARE_CLIMB
CLIMBING
RECOVER_FLAT
FAULT_STOP
```

状态切换依据包括：

- TOF 距离变化：判断是否接近楼梯或障碍。
- IMU 俯仰/横滚变化：判断是否进入爬升、是否姿态异常。
- 电机速度/扭矩反馈：判断是否卡住、打滑或负载异常。
- 三角轮角度反馈：确认执行机构是否到达目标位置。

关键动作阶段建议保留人工确认，避免早期测试阶段直接全自动执行。

### 第四阶段：数据记录与回放

建立实验数据记录能力，用于离线分析和状态机调参。

建议记录内容：

- 时间戳
- TOF 距离、状态、信号强度
- IMU 姿态、角速度、加速度
- 电机速度、扭矩、在线状态
- 三角轮角度、占空比
- 上位机状态机状态
- 下发给下位机的指令
- 异常事件和保护触发原因

通过日志回放，可以在不反复实机试错的情况下调整阈值和状态切换条件。

### 第五阶段：ROS2/C++ 迁移

当 Python 状态机逻辑稳定后，将核心决策逻辑迁移到 Linux C++/ROS2。

迁移目标：

- 将传感器数据抽象为 ROS2 Topic。
- 将状态机和策略决策实现为 C++ 节点。
- 保留 Python/FastAPI 仪表盘作为调试、监控和实验工具。
- 为后续接入相机、雷达或更多传感器留出接口。

### 长期目标

- 实现可靠的半自动爬楼助力。
- 提高楼梯识别和异常判断能力。
- 形成“上位机决策、下位机安全执行”的完整控制系统。
- 在保证安全边界的前提下，逐步提升自动化程度。

## 6. 风险与安全边界

| 风险 | 影响 | 当前策略 |
| --- | --- | --- |
| 上位机实时性不足 | 无法承担高频电机控制 | 底层闭环全部放在 STM32 |
| 串口通信中断 | 下位机可能继续执行旧目标 | 下位机必须实现指令超时保护 |
| 传感器误判 | 状态机可能错误切换 | 早期关键动作保留人工确认 |
| 姿态异常 | 机器人可能失稳 | IMU 异常应由下位机和上位机共同检测 |
| 参数调试风险 | 实机动作不可预期 | 先低速、空载、手动验证，再进入半自动 |

项目开发过程中，应始终遵守以下边界：

- 上位机可以下发目标，不直接控制底层 PWM。
- 下位机必须能在上位机失效时独立进入安全状态。
- 半自动状态机必须先通过日志回放和低速实机测试验证。
- 完全自主爬楼应作为长期目标，而不是当前阶段目标。

## 7. 阶段性验收标准

### 第二阶段验收标准

- [ ] 上位机能稳定读取真实下位机数据。
- [ ] Web 页面能实时展示 TOF、IMU、电机、三角轮状态。
- [ ] `/debug/state` 能反映串口连接、回包、CRC、超时等诊断信息。
- [ ] 上位机能手动下发基础目标参数。
- [ ] 下位机在上位机断连或指令超时时能进入安全状态。
- [ ] 手动动作测试中，机器人能完成低风险的模式切换、低速运动和三角轮动作验证。

### 第三阶段验收标准

- [ ] 能根据 TOF 和 IMU 数据进入接近楼梯状态。
- [ ] 能在人工确认后进入准备爬楼状态。
- [ ] 能根据三角轮和电机反馈判断动作是否执行到位。
- [ ] 姿态异常、通信异常或传感器异常时能进入 `FAULT_STOP`。

### 第四阶段验收标准

- [ ] 能记录完整实验日志。
- [ ] 能根据日志复盘状态机切换过程。
- [ ] 能通过离线数据调整阈值，减少实机试错次数。

## 8. 当前结论

当前项目最合适的路线是：继续使用 Python 上位机完成开发测试、数据可视化和手动动作验证；把实时控制、安全保护和底层电机闭环留给 STM32；当状态机和策略逻辑稳定后，再迁移到 Linux C++/ROS2 作为正式决策层。

这一阶段的核心不是追求完整自动化，而是先把通信、数据、手动控制和安全边界做稳。只有第二阶段联调稳定后，后续半自动爬楼状态机才有可靠基础。
